# Project Analysis

## Folder Structure

- `backend/`
  - `app.py`: FastAPI application, Google Sheets ingestion, SQLite schema creation, API endpoints.
  - `rail_dashboard.db`: SQLite database generated by the sync process.
  - `requirements.txt`: Python dependencies.

  - Generated analysis artifacts, images, logs, and exported workbook output.
- `Untitled spreadsheet.xlsx`
  - Local spreadsheet artifact.

## Backend Architecture

The backend is a single-file FastAPI service in `backend/app.py`.

Responsibilities:
- Pull raw data from Google Sheets CSV exports.
- Normalize rows into Python dictionaries.
- Rebuild a local SQLite database on startup and on demand.
- Expose read APIs for stations, catering units, sanctioned works, and earnings.
- Join related tables in SQL so the frontend receives enriched records.

Key implementation characteristics:
- Uses `requests` for Google Sheets export fetches.
- Uses `sqlite3` directly rather than an ORM.
- Uses startup-time sync to seed the database.
- Uses CORS middleware to allow the local frontend origins.
- Recreates the database tables during every sync.

Operational flow:
1. Fetch stations sheet.
2. Fetch units tab from the same sheet.
3. Fetch works sheet.
4. Fetch earnings tab from the stations sheet.
5. Parse and normalize each tab.
6. Build SQLite tables and linking tables.
7. Serve API data from SQLite.

## Frontend Architecture

The frontend is a Next.js App Router application.

Core structure:
- A single client page in `frontend/app/page.jsx`.
- Side navigation switches between Stations, Catering units, Earnings, and Sanctioned works.
- Search and filters are applied in-memory on fetched JSON.
- A fetch button explicitly re-syncs backend data from Google Sheets.
- A modal/detail drawer shows row-level data and linked entities.

Current UI behavior:
- Fetches all datasets on page load.
- Fetches again when the user presses the sync button.
- Keeps local React state for:
  - active view
  - search query
  - filters
  - selected row
  - loaded dataset snapshots
  - sync status

Design pattern:
- This is a dashboard shell, not a server-rendered data app.
- Data handling is client-heavy and derived from API responses after load.

## API Endpoints

Implemented in `backend/app.py`:

- `GET /api/health`
  - Returns a basic status check.
- `POST /api/sync`
  - Rebuilds the SQLite database from Google Sheets.
- `GET /api/stats`
  - Returns counts and summary metrics.
- `GET /api/stations`
  - Lists stations, with optional `q` and `category`.
- `GET /api/units`
  - Lists units, with optional `q` and `station_code`.
- `GET /api/works`
  - Lists sanctioned works, with optional `q`, `scope_type`, and `station_code`.
- `GET /api/earnings`
  - Lists earnings receipts, with optional `q`, `unit_no`, and `station_code`.

Notes:
- The API returns joined data for `units`, `works`, and `earnings`.
- `stations` is mostly a base table; the frontend derives richer views from joins and linked lists.

## Google Sheets Sync Flow

Sheets sources currently used:
- Stations / Units workbook:
  - `1UdRgQQPEkak1fUTuVH7jIn5R4sE3szAhM4VZJOdFIOU`
  - Stations tab
  - Units tab
  - Earnings tab
- Works workbook:
  - `1rJbfhcnEVuGMwGkT8yBObb9Bk5Hx0uU224EGxfplGRc`
  - `gid=590791228`

Sync logic:
1. The backend exports each tab as CSV using Google Sheets `gviz/tq`.
2. Rows are normalized by column header name.
3. Station codes are used as the primary station key.
4. Units link to stations through `unit.station_code = station.station_code`.
5. Earnings link to units through `earnings.unit_no = units.unit_no`.
6. Works are parsed from `Block Section Station`.
7. Work scopes are split into station/division/ABSS categories.
8. Parsed links are stored in mapping tables for fast joins.

Important behavior:
- The sync process currently drops and recreates the tables.
- The frontend explicitly triggers sync with a button.
- Startup also triggers a sync automatically.

## Data Models

There is no Pydantic model layer for the table entities. Data is handled as:
- raw CSV rows
- normalized Python `dict` objects
- SQLite rows converted to dictionaries for API output

Logical model relationships:
- `stations` is the root reference dataset.
- `units.station_code` points to `stations.station_code`.
- `earnings.unit_no` points to `units.unit_no`.
- `works` links to stations through parsed scope extraction.
- `earning_links` and `work_links` are derived helper tables.

Derived frontend entities:
- station detail view
- unit detail view
- work detail view
- earnings detail view
- linked station summaries for units, works, and earnings

## Security Issues

1. No authentication or authorization.
   - Any user who can reach the local backend can trigger sync and read the datasets.
2. Plain HTTP only.
   - The backend and frontend are served over localhost without transport security.
3. Google Sheets data is fetched directly from public export URLs.
   - There is no service-account isolation, credential management, or access control layer.
4. Broad CORS allowlist.
   - Multiple localhost origins are permitted.
5. Full table rebuild on sync.
   - There is no transactional versioning or rollback strategy if a fetch is partially broken.
6. No input validation on query parameters beyond simple string filtering.
   - This is not immediately SQL-injection-prone because parameters are bound, but it is still unguarded user input.
7. The frontend assumes the backend is trustworthy.
   - It renders many fields directly from sheet content without sanitization beyond React’s default escaping.

## Technical Debt

1. `backend/app.py` is doing too much.
   - Parsing, schema creation, sync orchestration, and API serving all live in one file.
2. Schema evolution is manual.
   - Any new sheet/tab requires code changes in multiple places.
3. `earnings` date parsing is weak.
   - Dates are treated as strings, which limits sorting and reporting quality.
4. Status logic is inferred in the UI.
   - “Active” vs “inactive” unit state is based on text matching.
5. Some source data uses inconsistent labels and spelling.
   - Examples: `RECIEPT TYPE`, station names, and mixed text fields.
6. Filter semantics are overloaded.
   - The same four filter labels mean different things across views.
7. The frontend duplicates a lot of derived logic.
   - Linking, aggregation, and display heuristics are computed client-side.
8. The project has accumulated generated assets and historical outputs in-repo.
   - Useful for analysis, but noisy for maintainability.

## Performance Bottlenecks

1. Full sheet re-download on every sync.
   - Large sheets will make sync latency grow quickly.
2. Full database rebuild on every sync.
   - Dropping and recreating tables is simple but expensive.
3. Frontend fetches all endpoints at once.
   - For larger datasets this becomes unnecessary network and memory load.
4. Client-side filtering over full dataset snapshots.
   - Search and filtering are done in the browser after loading everything.
5. Detail view aggregations are recomputed in React.
   - Per-station linked earnings and unit rollups are derived in the component tree.
6. Multiple joins are executed without indexes.
   - SQLite may remain acceptable at current size, but this will not scale cleanly.

## Recommendations

1. Split the backend into modules.
   - Separate parsing, schema, repository, and API layers.
2. Add typed data models.
   - Pydantic models for API responses and normalized sheet rows.
3. Persist parsed timestamps.
   - Convert dates to ISO format or native date columns.
4. Add indexes.
   - Especially on `units.station_code`, `earnings.unit_no`, `earnings.station_code`, and `work_links.station_code`.
5. Add incremental sync.
   - Rebuild only changed tabs or changed rows where possible.
6. Move derived rollups to the backend.
   - Station earnings totals, active units, and latest fee status should be API fields.
7. Standardize enum-like fields.
   - Normalize payment heads, receipt types, unit statuses, and station categories.
8. Narrow the frontend’s responsibilities.
   - Use the UI for presentation and interaction, not heavy data shaping.
9. Add error states and retry handling.
   - Especially for sync failures and sheet schema mismatches.
10. Add tests around parsing and linking.
   - Cover station parsing, unit linking, work scope parsing, and earnings joins.

## Bottom Line

The project is currently a functional dashboard stack built around:
- Google Sheets as source of truth
- FastAPI as the sync and query layer
- SQLite as local relational storage
- Next.js as the presentation layer

It works, but the architecture is still monolithic and sync-heavy. The next meaningful improvement is to move the derived relationships and summary metrics into the backend, then split the backend into smaller modules.
